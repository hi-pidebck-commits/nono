## 📄 개요 및 목적
본 보고서는 사용자와 AI 간의 질의응답(Q&A) 흐름에서 발생하는 '응답 파싱 실패' 현상의 근본적인 기술적 원인을 데이터 기반으로 리서치하고, 각 실패 유형별 진단과 시스템 개선 방안을 제시하는 것을 목적으로 합니다.

**핵심 문제 정의:** 응답 파싱 실패는 AI가 생성한 텍스트나 구조화된 데이터를 **최종 사용자 인터페이스(UI) 또는 후속 로직이 예상하는 특정 형식(스키마)**으로 성공적으로 변환하거나 해석하지 못할 때 발생합니다. 이는 LLM의 성능 문제가 아닌, 시스템 통합 단계에서의 구조적 문제인 경우가 많습니다.

## 🔍 1. 실패 원인의 3대 기술적 분류
파싱 실패는 크게 세 가지 계층(Layer)에서 기인하는 것으로 분석됩니다.

### A. API 호출 및 전송 계층 오류 (Technical Failure Point)
가장 낮은 수준의, 시스템 인프라와 관련된 문제들입니다.
*   **원인:**
    1.  **Rate Limiting/Quota 초과:** 짧은 시간 내에 너무 많은 API 호출을 시도하여 서비스 제공자 측에서 요청을 거부하는 경우 (429 Too Many Requests).
    2.  **Authentication Failure:** API 키 만료, 권한 부족 등으로 인해 백엔드 호출 자체가 실패할 때.
    3.  **Connection Timeout:** 네트워크 지연이나 서버 과부하로 인해 응답 데이터가 지정된 시간 내에 도착하지 못하는 경우.
*   **파싱 실패 메커니즘:** 데이터를 아예 받지 못하거나, 불완전한 헤더 정보만 받아와 파서를 작동시키지 못함.
*   **✅ 시스템 개선 방안 (Mitigation):**
    *   **Exponential Backoff 구현:** API 호출 실패 시 즉시 재시도하는 것이 아니라, 점진적으로 대기 시간을 늘려가며(예: 1초, 2초, 4초...) 재시도 로직을 의무화해야 합니다.
    *   **전용 에러 핸들링 계층:** API 레벨에서 발생하는 모든 예외를 사용자에게 보여주지 않고, '시스템 오류'라는 일반적인 메시지로 통일하여 노출해야 합니다.
